# Contexto geral para agentes de código

O projeto é um **Sistema Web de Controle de Uso da Sala de Informática da Biblioteca Central da UFAC**. O sistema substituirá o controle manual atualmente feito em papel, no qual os usuários informam matrícula, curso, número do computador, hora de entrada, hora de saída, espera e data. Esses registros devem alimentar relatórios diários, semanais, mensais e anuais de uso da sala.

O sistema deve ser construído pensando em três grandes objetivos: permitir que o próprio usuário da sala interaja diretamente com o sistema; permitir que a equipe da biblioteca acompanhe e administre o uso do espaço; e gerar relatórios consolidados automaticamente, especialmente por dia, turno, mês, ano, computador e quantidade de usuários.

Não considerar, nesta fase, os atores **Técnico de Manutenção / TI** e **Sistema de Autenticação Institucional**. O sistema terá autenticação própria ou fluxo simplificado interno, mas não deve ser modelado como integração externa institucional neste momento.

# Atores do sistema

## 1. Usuário da Sala

Representa qualquer pessoa autorizada a usar os computadores da Sala de Informática da Biblioteca Central da UFAC. Pode ser aluno, professor ou técnico-administrativo.

Esse ator interage diretamente com o sistema. Ele não depende do servidor da biblioteca para fazer o fluxo principal de uso. O próprio usuário deve conseguir logar, consultar computadores disponíveis, escolher um computador, visualizar horários disponíveis, agendar uso, registrar entrada, registrar saída, consultar ou entrar em fila de espera e solicitar troca de computador.

Especializações conceituais do ator:

Aluno.
Professor.
Técnico-administrativo.

Essas especializações podem aparecer no diagrama de casos de uso, mas no sistema elas tendem a compartilhar os mesmos fluxos principais.

## 2. Servidor da Biblioteca

Este ator reúne as responsabilidades que antes seriam divididas entre estagiário, atendente e servidor. Não haverá ator separado chamado “Estagiário” nesta fase.

O Servidor da Biblioteca acompanha a operação da sala, consulta sessões ativas, acompanha fila de espera, acompanha reservas, corrige registros quando necessário, registra ocorrências administrativas ou técnicas simples, controla a situação dos computadores e gera relatórios operacionais.

Ele também pode realizar ações manuais em casos excepcionais, como registrar entrada/saída para um usuário que teve dificuldade, corrigir horário lançado incorretamente ou liberar computador quando houver inconsistência.

## 3. Supervisor da Biblioteca

É o ator de visão gerencial. Ele acompanha indicadores, valida relatórios consolidados, consulta dados históricos e analisa a utilização da sala por turno, dia, mês, ano, computador e perfil de usuário.

O Supervisor da Biblioteca não é o ator principal dos registros cotidianos de entrada e saída. Seu foco é análise, conferência, tomada de decisão e acompanhamento institucional.

## 4. Administrador do Sistema

É o ator com maior permissão administrativa. Ele gerencia contas, permissões, perfis de acesso, computadores, parâmetros do sistema, turnos, regras de agendamento e logs.

O Administrador do Sistema pode configurar dados estruturais do sistema, mas não deve ser confundido com o usuário comum da sala nem com o supervisor gerencial.

# Fluxo principal do Usuário da Sala

O fluxo principal do usuário deve seguir esta lógica:

1. O usuário acessa o sistema e realiza login.

2. Após o login, o usuário escolhe a data de uso/agendamento. As opções permitidas nesta fase são apenas:

Hoje.
Amanhã.

O sistema não deve permitir agendamento para datas livres no futuro além do dia seguinte. Também não deve permitir horários passados no dia atual.

3. Depois de escolher “Hoje” ou “Amanhã”, o usuário visualiza os computadores disponíveis para aquele dia. Computadores em manutenção não devem aparecer como disponíveis para uso.

4. O usuário seleciona um computador.

5. Após selecionar o computador, o sistema exibe os horários disponíveis para aquele computador na data escolhida.

6. O usuário escolhe um horário disponível.

7. O sistema confirma o agendamento ou uso.

8. No momento de usar a máquina, o usuário registra sua entrada.

9. Durante o uso, a sessão fica ativa e o computador fica ocupado.

10. O usuário pode solicitar troca de computador. Nesse caso, o sistema deve mostrar apenas computadores disponíveis e que não estejam em manutenção.

11. Ao trocar de computador, o sistema deve manter o histórico da sessão, registrando que o usuário iniciou em um computador e depois passou para outro.

12. Ao finalizar, o usuário registra sua saída.

13. O sistema calcula automaticamente o tempo de permanência, libera o computador e atualiza os dados que alimentarão os relatórios.

# Fluxos alternativos importantes

## Ausência de computadores disponíveis

Quando não houver computador disponível para o dia/horário selecionado, o sistema deve permitir que o usuário consulte a fila de espera e entre nela.

A fila de espera deve registrar pelo menos: usuário, data, horário de entrada na fila, posição, status e eventual computador desejado, caso o sistema permita preferência por computador.

## Entrada em fila de espera

O usuário pode entrar na fila quando não houver computador disponível ou quando todos os horários desejados estiverem ocupados.

A fila deve ser ordenada por horário de entrada ou por regra definida pela biblioteca. A regra padrão recomendada é ordem cronológica de entrada.

## Saída da fila

O usuário pode sair voluntariamente da fila. O Servidor da Biblioteca também pode remover uma entrada da fila quando houver erro, desistência presencial ou inconsistência.

## Troca de computador

A troca de computador deve ser permitida durante uma sessão ativa.

Regra importante: não sobrescrever simplesmente o número do computador antigo. O sistema deve preservar histórico. Recomenda-se modelar a sessão de uso separada do histórico de computadores usados na sessão.

Exemplo:

Sessão 123: usuário X, início 09:00, fim 11:00.
Uso de computador 1: PC-03, das 09:00 às 10:00.
Uso de computador 2: PC-07, das 10:00 às 11:00.

Isso evita perda de informação nos relatórios e permite saber quais máquinas foram realmente usadas.

## Usuário não comparece ao agendamento

O sistema deve prever status de reserva expirada ou não comparecida. O tempo de tolerância deve ser configurável. Caso ainda não exista regra institucional, usar uma configuração simples no sistema, por exemplo 10 ou 15 minutos, mas deixar isso como parâmetro editável pelo Administrador.

## Computador em manutenção

Computador em manutenção não pode ser reservado, não pode receber entrada de usuário e não deve aparecer como disponível.

Se um computador for colocado em manutenção enquanto houver reservas futuras, o sistema deve impedir novos agendamentos e sinalizar as reservas afetadas para o Servidor da Biblioteca.

# Regras de negócio principais

O usuário precisa estar logado para consultar computadores, agendar, entrar em fila, registrar entrada, registrar saída ou trocar de computador.

O usuário só pode ter uma sessão ativa por vez.

Um computador só pode ter uma sessão ativa por vez.

Um computador em manutenção nunca deve ser considerado disponível.

O sistema só deve permitir agendamento para hoje ou amanhã.

Para o dia atual, o sistema não deve permitir seleção de horários já passados.

O sistema deve calcular automaticamente o turno de uso a partir da hora de entrada.

Turnos observados no modelo de relatório da biblioteca:

1º turno: 7h15 às 13h.
2º turno: 13h às 17h.
3º turno: 17h às 21h.

Esses horários devem ser configuráveis pelo Administrador, não hardcoded de forma permanente.

A saída do usuário encerra a sessão, calcula a duração e libera o computador.

A troca de computador não encerra necessariamente a sessão geral; ela encerra o vínculo com o computador anterior e cria novo vínculo com o computador novo.

Os relatórios devem ser gerados a partir dos registros reais de sessão, e não preenchidos manualmente.

A coluna “espera” do modelo antigo deve ser representada no sistema pela existência ou não de entrada na fila de espera, tempo de espera ou status de espera.

O sistema deve gerar relatórios com base nas sessões finalizadas e, quando necessário, nas reservas e filas.

Relatórios mínimos:

Relatório diário de uso.
Relatório semanal de uso.
Relatório mensal de uso.
Relatório anual de uso.

Agrupamentos importantes:

Por dia.
Por turno.
Por computador.
Por curso ou setor.

# Estados importantes

## Estado do computador

DISPONIVEL.
OCUPADO.
MANUTENCAO.
INATIVO.

Transições:

DISPONIVEL → OCUPADO, quando uma sessão começa.
OCUPADO → DISPONIVEL, quando a sessão termina.
DISPONIVEL → MANUTENCAO, quando o servidor/admin marca manutenção.
MANUTENCAO → DISPONIVEL, quando a manutenção termina.
DISPONIVEL → INATIVO, quando o computador deixa de ser usado no sistema.

## Estado da reserva

AGENDADA.
EM_USO.
CONCLUIDA.
CANCELADA.
EXPIRADA.
NAO_COMPARECEU.

Transições:

AGENDADA → EM_USO, quando o usuário registra entrada.
EM_USO → CONCLUIDA, quando o usuário registra saída.
AGENDADA → CANCELADA, quando o usuário ou servidor cancela.
AGENDADA → EXPIRADA ou NAO_COMPARECEU, quando passa o tempo de tolerância.

## Estado da sessão

ATIVA.
FINALIZADA.
CANCELADA.

Transições:

ATIVA → FINALIZADA, quando o usuário registra saída.
ATIVA → CANCELADA, em caso excepcional administrativo.

# Requisitos não funcionais

O sistema deve usar controle de acesso por perfil.

O sistema deve registrar logs das ações administrativas sensíveis.

O sistema deve coletar apenas os dados necessários para controle de uso da sala.

O sistema deve proteger dados pessoais com autenticação, autorização e armazenamento seguro.

O sistema deve evitar exposição pública de matrícula, histórico de uso ou dados pessoais.

O sistema deve ter interface simples, responsiva e acessível.

O sistema deve funcionar bem em computadores da própria biblioteca e em dispositivos móveis, caso o usuário acesse pelo celular.

O sistema deve permitir exportação de relatórios em PDF, CSV ou planilha.

O sistema deve priorizar regras configuráveis em vez de valores fixos, especialmente turnos, horários de funcionamento e tolerância de atraso.

# Restrições explícitas para o agente de código

Não criar ator Técnico de Manutenção / TI nesta fase.

Não criar ator Sistema de Autenticação Institucional nesta fase.

Não tratar o Estagiário como ator separado; as ações dele devem estar dentro do ator Servidor da Biblioteca.

Não assumir que o Servidor da Biblioteca fará o fluxo principal pelo usuário. O Usuário da Sala interage diretamente com o sistema.

Não permitir agendamento para datas além de hoje ou amanhã.

Não permitir computador em manutenção como opção disponível.

Não sobrescrever histórico quando houver troca de computador.

Não gerar relatórios com dados manuais desconectados das sessões de uso.

Não hardcodar os turnos de forma definitiva; usar configuração.

Não expor dados pessoais sem necessidade.
